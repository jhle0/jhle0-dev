머신러닝에서 **가장 중요한 것은 데이터**라고 할 수 있다.

문제가 될 수 있는 데이터와 모델들을 알아보자.

### 충분하지 않은 양의 훈련 데이터

- 대부분의 머신러닝은 데이터가 많아야 좋다

### 대표성 없는 훈련 데이터

일반화가 잘되려면 훈련 데이터가 일반화하고 싶은 새로운 사례를 잘 대표하는 것이 중요하다.

- 샘플이 작으면 → 샘플링 노이즈가 생기고
- 샘플이 매우 커도 표본 추출 방법이 잘못되면 대표성을 띠지 못할 수 있다

→ 이를 **샘플링 편향(sampling bias)** 이라고 한다

### 낮은 품질의 데이터

훈련 데이터가 오류, 이상치, 잡음으로 가득하다면 머신러닝은 제대로 작동하지 못한다.

→ **훈련 데이터 정제**에 시간을 많이 투자해야 한다.

- 예시 :
    - 일부 샘플이 이상치라는게 명확하면 해당 샘플을 무시하거나 수동으로 고쳐야한다.
    - 일부 샘플에 특성 몇 개가 빠져있다면
        - 이 특성을 무시하거나, 이 샘플을 무시하거나, 빠진 값을 채우거나 해야한다.

### 관련 없는 특성

**garbage in, garbage out** 이라는 말이 있다.

즉, 훈련에 사용할 좋은 특성들을 찾아야 한다.

→ 이 과정을 특성 공학(feature engineering)이라고 한다.

- 특성 선택(feature selection) : 가장 유용한 특성을 선택한다
- 특성 추출(feature extraction) : 특성을 결합하여 더 유용한 특성을 만든다
- 데이터 수집 : 새로운 데이터를 수집해 새 특성을 만든다

### 훈련 데이터 과대적합(overfitting)

사람은 종종 과도하게 일반화를 할 때가 있다.

머신러닝에서는 이를 **과대적합(overfitting)** 이라고 한다.

→ 모델이 훈련 데이터에는 너무 잘 맞지만 일반성이 떨어진다는 뜻이다.

과대적합은 훈련 데이터의 양과 잡음에 비해 **모델이 너무 복잡할 때** 일어난다

- 해결 방법 :
    - 파라미터 수가 적은 모델을 선택
    - 특성 수를 줄이거나, 모델에 제약을 가하여 단순화
    - 훈련 데이터를 더 많이 모은다
    - 잡음을 줄인다

### 규제, 자유도, 하이퍼파라미터

모델을 단순하게 하고 과대적합의 위험을 줄이기 위해 모델에 제약을 가하는 것 을

→ **규제(regularizaion)** 라고 한다.

예를 들어 선형 모델은 두 개의 모델 파라미터 $\theta_0$과 $\theta_1$을 가지고 있다.

이는 두 개의 **자유도(degree of freedom)** 를 부여한 것이다.

$\theta_1 = 0$ 이 되도록 강제하면 알고리즘에는 한 개의 자유도만 남게 되는 것이다.

학습하는 동안 적용할 규제의 양은 → **하이퍼파라미터(hyperparameter)** 가 결정

하이퍼파라미터를 매우 큰 값으로 설정하면

→ 과대적합될 가능성은 없겠지만 좋은 모델 X

### 과소적합(underfitting)

과소적합은 과대적합의 반대이다.

즉, 모델이 너무 단순해서 데이터의 내재된 구조를 학습하지 못할 때 일어난다.

- 해결 방법 :
    - 모델 파라미터가 더 많은 강력한 모델 선택
    - 더 좋은 특성을 제공
    - 모델의 제약을 줄인다(하이퍼파라미터를 감소시킨다)

## 테스트와 검증

모델이 새로운 샘플에 얼마나 잘 일반화될지 알 수 있는 방법은

새로운 샘플에 실제로 적용해보는 것이다.

이를 위해 훈련 데이터를 **훈련 세트**와 **테스트 세트** 두 개로 나눈다.

훈련 세트를 사용해 모델을 훈련 → 테스트 세트를 사용해 모델을 테스트

이때 새로운 샘플에 대한 오차 비율을 **일반화 오차(generalization error)** 라고 한다.

훈련 오차가 작지만, 일반화 오차가 크다면 이는 모델이 훈련 데이터에 과대적합되었다는 뜻이다.

### 하이퍼파라미터 튜닝과 모델 선택

모델 평가는 간단하다. 테스트 세트를 사용하면 된다.

테스트 세트를 사용해 더 잘 일반화 되는 모델을 찾은 뒤 이제 규제를 적용해보자.

이때 많은 하이퍼파라미터 값으로 모델을 테스트 해본다음 가장 일반화 오차가 낮은 모델을 찾았다고 해보자.

하지만 이 모델은 성능이 낮았다.

왜 그럴까?

이는 모델과 하이퍼파라미터가 테스트 세트에 최적화된 모델을 만들었기 때문이다.

→ 이를 해결하기 위해 **홀드아웃 검증(holdout validation)** 을 사용한다

### **홀드아웃 검증(holdout validation)**

홀드아웃 검증은

훈련 세트의 일부를 떼어내어 여러 후보 모델을 평가하고 가장 좋은 하나를 선택한다

이 새로운 홀드아웃 세트를 **검증 세트(validation set)** 라고 한다. (개발 세트 development set 라고도 한다)

이제 훈련세트에서 다양한 하이퍼 파라미터 값을 가진 여러 모델을 훈련한다

다음 검증 세트에서 가장 높은 성능을 내는 모델을 선택한다.

가장 높은 성능을 내는 모델을 선택했으면, 이제 전체 훈련 세트(검증 세트를 포함한)에서 다시 훈련하여 최종 모델을 만든다

마지막으로 최종 모델을 테스트 세트에서 평가하여 일반화 오차를 추정한다.

데이터 셋을 다시 정리해보면

### Train(훈련셋)

- **목적**: 모델 파라미터(가중치)를 학습하는 데이터.
- **사용**: loss를 최소화하도록 반복 학습에 직접 사용.
- **주의**: Train 성능이 높아도 일반화 성능이 좋다는 보장은 없다(과적합 가능).

### Validation(검증셋, Val/Dev)

- **목적**: **모델/하이퍼파라미터 선택**을 위한 데이터.

### Test(테스트셋)

- **목적**: 최종 성능을 **한 번** 측정하는 “최종 평가용” 데이터.
- **원칙**:
    - 모델/튜닝/전처리 결정에 **절대 사용하지 않는다**.
    - 여러 번 테스트셋을 확인하면 사실상 검증셋처럼 써버리는 것이고, 성능이 부풀려진다.
